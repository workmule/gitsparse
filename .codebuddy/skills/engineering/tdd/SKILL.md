# 测试驱动开发

## 理念

**核心原则**：测试应通过公共接口验证行为，而非实现细节。代码可以完全改变；测试不应变。

**好的测试**是集成风格的：它们通过公共 API 执行真实代码路径。它们描述系统*做什么*，而非*怎么做*。好的测试读起来像规格说明——"用户可以用有效购物车结账"准确告诉你存在什么能力。这些测试能在重构后存活，因为它们不关心内部结构。

**坏的测试**与实现耦合。它们 mock 内部协作者、测试私有方法、或通过外部手段验证（如直接查询数据库而非使用接口）。警告信号：你重构时测试坏了，但行为没变。如果你重命名一个内部函数而测试失败，这些测试测的是实现，不是行为。

参见 [tests.md](tests.md) 获取示例，[mocking.md](mocking.md) 获取 mock 指南。

## 反模式：水平切片

**不要先写所有测试，再写所有实现。** 这是"水平切片"——把 RED 当成"写所有测试"，GREEN 当成"写所有代码"。

这会产出**垃圾测试**：

- 批量写出的测试测试的是*想象的*行为，而非*实际的*行为
- 你最终测试的是事物的*形状*（数据结构、函数签名）而非面向用户的行为
- 测试对真实变化不敏感——行为坏了它们通过，行为没问题它们反而失败
- 你跑在车灯前面，在理解实现之前就锁定了测试结构

**正确做法**：通过 tracer bullet 做垂直切片。一个测试 → 一个实现 → 重复。每个测试响应于你从上一个循环学到的东西。因为你刚写了代码，你精确知道哪些行为重要以及如何验证。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域术语表，让测试名和接口词汇匹配项目语言，并尊重你涉及区域的 ADR。

在写任何代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认测试哪些行为（排优先级）
- [ ] 识别[深模块](deep-modules.md)的机会（小接口，深实现）
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

问："公共接口应该是什么样的？哪些行为最重要需要测试？"

**你不可能测试所有东西。** 与用户确认哪些行为最重要。把测试精力集中在关键路径和复杂逻辑上，而非每个可能的边界情况。

### 2. 示踪弹（Tracer Bullet）

写一个测试确认系统的一件事：

```
RED:   为第一个行为写测试 → 测试失败
GREEN: 写最少代码让测试通过 → 测试通过
```

这是你的示踪弹——证明路径端到端工作。

### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少代码让它通过 → 通过
```

规则：

- 一次一个测试
- 只写足够通过当前测试的代码
- 不要预期未来的测试
- 保持测试聚焦于可观察行为

### 4. 重构

所有测试通过后，寻找[重构候选](refactoring.md)：

- [ ] 提取重复
- [ ] 加深模块（把复杂性移到简单接口背后）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 思考新代码揭示了关于现有代码的什么
- [ ] 每个重构步骤后运行测试

**RED 状态时绝不重构。** 先到 GREEN。

## 每轮检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试只使用公共接口
[ ] 测试能在内部重构后存活
[ ] 代码对当前测试是最少的
[ ] 没有添加推测性功能
```
